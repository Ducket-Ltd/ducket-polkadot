  <files>frontend/src/hooks/usePurchaseTicket.ts</files>
  <action>
Create a custom hook that encapsulates the full purchase flow for both DOT and USDC paths.

**Exported types:**
- `PurchaseStep = 'idle' | 'approving' | 'approve-confirming' | 'purchasing' | 'purchase-confirming' | 'success' | 'error'`
- `PaymentMethod = 'DOT' | 'USDC'`

**Hook signature:**
```typescript
export function usePurchaseTicket(
  tokenId: number | null,
  quantity: number,
  paymentMethod: PaymentMethod,
  tier: MergedTier | undefined
): {
  step: PurchaseStep
  needsApprove: boolean
  stepLabel: string        // "Step 1/2: Approve USDC" or "Step 2/2: Purchase Ticket" or "Purchase Ticket"
  isPending: boolean       // true when waiting for wallet or chain confirmation
  errorMessage: string
  isSuccess: boolean
  execute: () => void
  reset: () => void
}
```

**Implementation details:**
1. Use `useAccount()` to get `address`. Guard all writes with `if (!address) return`.
2. Use `useReadContract` to check USDC allowance — `MOCK_USDC_ABI.allowance(address, CONTRACT_ADDRESS)` with `query: { enabled: !!address && paymentMethod === 'USDC' }`.
3. Compute `needsApprove = paymentMethod === 'USDC' && (!allowance || allowance < totalStablePrice)` where `totalStablePrice = tier.stablePrice * BigInt(quantity)`.
4. Use a SINGLE `useWriteContract` instance (not two). Reuse it for both approve and purchase calls.
5. Use `useWaitForTransactionReceipt({ hash: txHash })` to wait for confirmations.
6. State transitions via `useEffect` on `isConfirmed`:
   - If `step === 'approve-confirming'` and confirmed: `refetchAllowance()`, then call `executePurchase()`.
   - If `step === 'purchase-confirming'` and confirmed: `setStep('success')`.
7. Handle `writeError`: set `step` to `'error'`, capture `writeError.message` (or short version).
8. `stepLabel` logic:
   - DOT path: always `"Purchase Ticket"`
   - USDC path with needsApprove and step is idle/approving/approve-confirming: `"Step 1/2: Approve USDC"`
   - USDC path when purchasing/purchase-confirming: `"Step 2/2: Purchase Ticket"`
9. `reset()`: set step to 'idle', clear error.
10. DOT path: `writeContract({ ...DUCKET_ABI, functionName: 'mintTicket', args: [BigInt(tokenId), address, BigInt(quantity)], value: tier.price * BigInt(quantity) })`.
11. USDC approve: `writeContract({ ...MOCK_USDC_ABI, functionName: 'approve', args: [CONTRACT_ADDRESS, totalStablePrice] })`.
12. USDC purchase: `writeContract({ ...DUCKET_ABI, functionName: 'mintTicketWithToken', args: [BigInt(tokenId), address, BigInt(quantity)] })`.

**Important — avoid these anti-patterns (from research):**
- Do NOT create two separate `useWriteContract` instances — stale closure issues. Use one, reused for approve then purchase.
- Do NOT use `parseUnits` on stablePrice — it is already in 6-decimal units from the contract. Using parseUnits would double-scale.
- Cast all addresses as `0x${string}` for wagmi type safety.
- Include `query: { enabled: !!address }` on the allowance read.

**Note on useEffect for step transitions:** The single `useWriteContract` reuses its `data` (hash) field. When approve confirms and we call `writeContract` again for purchase, the hash changes. Use a `useEffect` watching `isConfirmed` + current `step` to drive transitions. Add a ref or state flag to track which transaction (approve vs purchase) the current hash belongs to, since `useWaitForTransactionReceipt` always watches the latest hash from `useWriteContract`. The simplest approach: after approve is confirmed, immediately call the purchase writeContract — the new hash replaces the old one and useWaitForTransactionReceipt watches the new one.
